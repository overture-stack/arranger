Fixed a real design gap in `modules/sqon` surfaced by a Lyric-side cost estimate for the sqon-builder migration, and closed two smaller documentation gaps flagged in the same review.

- Lyric's migration-cost estimate flagged a concrete trap: the schema accepts operator aliases (`=`, `>=`, `filter`, etc.) as valid input, but nothing normalized them, so code switching on a parsed node's `.op` could accept a query using an alias and then fail to match any canonical branch. Confirmed by reading the source directly: `normalizeSqonOp` existed but was never called anywhere inside the package, and `InLikeOpSchema`/`RangeLikeOpSchema` (`schema/constants.ts`) validate canonical-or-alias without a `.transform()`, so `SqonSchema.parse()` and `SqonBuilder.from()` both returned aliases unchanged.
- Fixed at the source rather than documented as a footgun: added `builder/normalize.ts` (`normalizeSqonNode`, recursive, pure) and wired it into `SqonBuilder.from()`, so every leaf's `op` is guaranteed canonical after parsing, recursively through nested combinations. Deliberately did not add the transform to the schema definitions themselves (`InLikeFilterSchema` etc.): those same schema objects are fed directly into `zodToJsonSchema()` for `getSqonJsonSchema()`, and a schema-level transform would have changed or broken the generated JSON Schema's `op` enum. Verified post-fix that `getSqonJsonSchema()`'s output is byte-for-byte unaffected (still lists both canonical and alias values, as before).
- `SqonSchema.parse()` called directly (bypassing the builder) still returns aliases unnormalized; documented this explicitly in `README.md` and `docs/sqon-builder-absorption.md`, since Lyric's own report had proposed exactly this bypass (`parseSQON` calling `SqonSchema.parse()` directly for the "just validate" case) without knowing about the normalization gap. Corrected that recommendation in the absorption doc: prefer `SqonBuilder.from(raw).toValue()`, or call the newly-exported `normalizeSqonNode()` if bypassing the builder for another reason.
- Exported two structural type guards that already existed internally but were never in the public surface: `isGroupNode`, `isFieldFilter` (`builder/utils.ts`). These are the direct equivalent of `sqon-builder`'s `isCombination`/array-filter-style guards that the absorption doc's migration section said weren't ported; they were already implemented, just not exported. Documented in a new "Type guards" table in `README.md`.
- Documented the previously-unmentioned `pivot` field (present on every leaf and combination node, an ES/OpenSearch nested-path concept) in `README.md`'s Notes and the absorption doc: safe to ignore for a non-ES/OS consumer, not an oversight.
- Root `CHANGELOG.md`'s existing "SQON operators" entry claimed `filter` "normalizes to `wildcard` at query-build time," which wasn't actually true until this fix (nothing normalized anything, for any alias, anywhere). Added a bullet documenting the actual fix and the new exports; the existing wildcard-specific line is now accurate rather than aspirational.
- Tests: `builder/normalize.test.ts` (new, 7 cases: each alias family, recursion through nested combinations, non-mutation) plus 2 new integration-level cases in `builder/index.test.ts` exercising `SqonBuilder.from()` end-to-end. Full `modules/sqon` suite: 119/119 passing. Full package build (`npm run build -w modules/sqon`) verified clean, including the DTS step, after fixing a type error the first cut of `normalizeSqonNode` introduced (`normalizeSqonOp`'s return type is the full canonical-op union, not narrowed to leaf ops; resolved with the same `as unknown as SqonNode` pattern already used elsewhere in this file for equivalent cases). `npm run test:dev`'s CJS import-integration check confirms the new exports load correctly from the built package.
- Not touched: extending real support for the 7 operators `sqon-builder` never had (`notIn`, `someNotIn`, `all`, `gte`, `lte`, `between`, `wildcard`) in any consumer; Lyric's own report already scoped that as a separate, larger effort from the same-behaviour migration.

Everything above is unstaged in the working tree; nothing has been committed.

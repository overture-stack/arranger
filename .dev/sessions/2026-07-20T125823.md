Investigated and closed an external HCMI pentest finding on GraphQL array-based query batching, plus one bundled fix picked during scoping.

- Confirmed the finding: both `ApolloServer` instances in `graphqlRoutes.ts` (`createEndpoint`) never set `allowBatchedHttpRequests`, so `apollo-server-core@3.13.0` left it at its permissive default and processed any POSTed JSON array as independent parallel operations. `maxAliasesRule`/`maxDepthRule` cap the cost of each operation in a batch but not the number of operations, so this was a genuine, distinct DoS vector from the ones those rules already close.
- Added a new `enableGraphQLBatching` feature flag, wired through the same chain as `disableFilters`/`disableDownloads`/`disablePlayground` (plain env parsing, no `NODE_ENV` coupling) but inverted and disabled by default: Arranger has no legitimate internal use for HTTP-level batching, so it's opt-in rather than opt-out, matching `enableSets`'s polarity rather than the `disable*` cluster. Initially shipped as `disableGraphQLBatching` (default `false`, batching allowed unless disabled); rewired to `enableGraphQLBatching` (default `false`, batching rejected unless enabled) once flagged that batching has no current use, so secure-by-default is strictly better here. Wired through `modules/types/src/configs/constants.ts`, `apps/search-server/src/configs/fromEnv/localEnvs.ts` + `aggregator.ts`, both `ApolloServer` configs and the `createEndpoint`/`arrangerRoutes` plumbing in `graphqlRoutes.ts`, `configs.json.schema`, `.env.schema`, and the README feature-flags table. `createEndpoint` defaults the parameter itself (`enableGraphQLBatching = false`) so any caller omitting it still gets `allowBatchedHttpRequests: false` explicitly, since Apollo only blocks on strict `=== false`, not `undefined`.
- Bundled in the GraphQL field-name-suggestion leak fix (picked from a broader vulnerability survey done before implementing): added a custom `formatError` to both `ApolloServer` configs stripping the `Did you mean "x"?` suffix graphql-js appends to validation errors, which leaked schema structure on a code path separate from introspection. First implementation called graphql-js's `formatError`/`error.toJSON()`, which crashed on the batching-rejection error (a plain `Error`, no `GraphQLError` prototype); fixed by building the formatted shape from the enumerable fields Apollo's `enrichError` already normalizes onto the error object, no `toJSON` dependency.
- Found and fixed a real bug while tracing `disableGraphQLIntrospection`'s wiring as the template: `stringToBool`'s `str = ''` default parameter meant its `str === undefined` branch was unreachable dead code, so the `?? isProd` / `stringToBool(x, isProd)` fallback never fired at any call site. `disableGraphQLIntrospection` has silently defaulted to `false` (open) in every environment regardless of `NODE_ENV`, contrary to what the roadmap and README both claimed. Fixed in `modules/types/src/tools/stringFns.ts` (removed the default on `str`) with new tests in `stringFns.test.ts`. This is a real behaviour change: production deployments that relied on the old (buggy) open-by-default behaviour, particularly network-aggregation remote nodes that must keep introspection enabled per the existing README caveat, need `DISABLE_GRAPHQL_INTROSPECTION=false` set explicitly before this ships. Noted in `.dev/roadmap.md`'s GraphQL server migration section.

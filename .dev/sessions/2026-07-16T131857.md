Fixed a production GraphQL validation bug in Arranger Charts and a mirror bug in the graphql-router remote node query builder (both from the network aggregation PR), added regression tests, wired up unit testing for `modules/charts`, as well as type fixes to cover integration experience.

- `modules/charts/src/query/generateCharts.ts`: `$nodesFilter` is now declared only when the `network` block is present; it was previously always declared, failing GraphQL validation ("Variable $nodesFilter is never used") for any chart not using network functionality. Also fixed the `fieldQueries` reduce silently discarding all previously accumulated fields when one had an unsupported `gqlTypename` (pre-existing, not introduced by the network PR).
- `modules/graphql-router/src/network/resolvers/query.ts`: `$filters` is now always declared in `createRemoteNodeGQLQuery`; it was previously omitted whenever no aggregation fields were requested, failing GraphQL validation ("Variable $filters is not defined") for network-only, no-aggregation queries.
- `modules/charts/package.json`: added a `test` script (`tsx --test`) and `tsx` devDependency, matching the `node:test`/`assert` convention already used by `modules/sqon`, `modules/types`, and `modules/graphql-router`.
- `modules/charts/src/query/generateCharts.test.ts`, `modules/graphql-router/src/network/resolvers/query.test.ts`: new regression tests covering both fixes; all pass.
- `modules/components/src/DataContext/types.ts`: added `networkNodesFilter`/`setNetworkNodesFilter` to `DataContextInterface`; the context already provided both at runtime, the public type just didn't declare them, so consumers got no type-safety on either.
- `modules/charts/src/components/charts/Bar/View.tsx`: removed a duplicate `fieldName` property from `BarChartViewProps`.
- `.dev/tech-debt.md`: removed the two now-resolved entries (missing test coverage, reduce bug); trimmed `query.ts` out of the broader "no unit tests for network resolvers" entry's remaining scope.
